## Why a Separate Gateway?

Agents interact with external tools via ad-hoc API calls with direct access to credentials. If an agent leaks a key -- through prompt injection, a bad tool call, or a context dump -- that key is live until someone manually rotates it. The median time from credential leak to first malicious use is **5 minutes**.

CommandClaw-MCP solves this with a proxy architecture:

```
Agent --> (phantom token) --> commandclaw-mcp --> (real credentials) --> External MCP Servers
```

The agent never holds real API keys. Even if the phantom token leaks, it expires within the hour and only works through the gateway.

## Quick Start (Docker Compose)

**Prerequisites:** Docker with Compose v2, Python 3.12+

```bash
# 1. Clone and enter the repo
gh repo clone FnSK4R17s/commandclaw-mcp && cd commandclaw-mcp

# 2. Generate config and .env with a random encryption seed
make setup

# 3. Add your MCP servers to the config
#    Edit ~/.commandclaw/mcp.json — see "Configuration" below

# 4. Start the stack (gateway + Redis + Cerbos)
make up
```

The gateway is now running at **http://localhost:8420**.

```bash
make health      # Check health + readiness
make logs        # Follow container logs
make down        # Stop everything
```

## Local Development

Run the gateway directly on your machine with only Redis and Cerbos in Docker:

```bash
# Install the package in editable mode
make install

# Start Redis + Cerbos containers
make dev-deps

# Run the gateway (connects to localhost Redis + Cerbos)
make dev
```

## Configuration

The gateway reads from **`~/.commandclaw/mcp.json`** (outside the repo, never in Git). Environment variables with the `COMMANDCLAW_` prefix override JSON values.

`make setup` generates a starter config with a random encryption seed. Add your upstream MCP servers and agent access rules:

```json
{
  "gateway": {
    "host": "127.0.0.1",
    "port": 8420,
    "encryption_seed": "<auto-generated>"
  },
  "servers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "ghp_..." }
    },
    "notion": {
      "command": "npx",
      "args": ["-y", "@notionhq/notion-mcp-server"],
      "env": { "NOTION_API_KEY": "ntn_..." }
    }
  },
  "access": {
    "coding-agent": {
      "roles": ["developer"],
      "tools": ["github", "notion"],
      "rate_limit": { "requests_per_minute": 60 }
    },
    "research-agent": {
      "roles": ["reader"],
      "tools": ["notion"],
      "rate_limit": { "requests_per_minute": 30 }
    }
  },
  "redis": { "url": "redis://localhost:6379/0" },
  "cerbos": { "host": "localhost", "port": 3593 }
}
```

Servers can be **stdio** (`command` + `args`) or **HTTP** (`url`). The gateway bridges stdio servers to HTTP automatically.

### Environment Variable Overrides

Env vars use `COMMANDCLAW_` prefix with `__` for nesting. They **override** JSON config values:

```bash
COMMANDCLAW_GATEWAY__PORT=9000              # Override gateway port
COMMANDCLAW_REDIS__URL=redis://myhost:6379  # Override Redis URL
COMMANDCLAW_CERBOS__HOST=cerbos-pdp         # Override Cerbos host
```

Docker Compose sets these automatically for container networking. See `.env.example` for the full list.

### Config Path Override

Set `COMMANDCLAW_CONFIG` to load config from a different path:

```bash
COMMANDCLAW_CONFIG=/etc/commandclaw/mcp.json commandclaw-mcp
```

## API Endpoints

### Health & Operations

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/health` | GET | Liveness probe |
| `/ready` | GET | Readiness probe (Redis, Cerbos, rotation manager) |
| `/metrics` | GET | Prometheus metrics |

### Session Management

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/sessions` | POST | Create a phantom session: `{"agent_id": "coding-agent"}` |
| `/token` | GET | Poll for current token (authenticated) |
| `/sessions/{token}` | DELETE | Revoke a session immediately |

### Agent Integration

Agents only need two values:

```
COMMANDCLAW_MCP_GATEWAY=http://localhost:8420
COMMANDCLAW_MCP_KEY=<phantom_token from /sessions>
```

## Three Security Layers

### 1. Phantom Token Pattern (Credential Isolation)

Agents receive opaque, meaningless tokens. Real API keys live only in the gateway's encrypted vault.

| What the agent sees | What the gateway holds |
|---------------------|----------------------|
| `dG9rZW5fYWJj...` (random 256-bit) | `ghp_realGitHubToken...` (encrypted at rest) |

- HMAC-SHA256 signed requests using the phantom token as the signing key
- Fernet + Argon2id encryption at rest
- Hourly rotation with 5-minute dual-key overlap window
- Instant revocation via session DELETE

### 2. Per-Agent Tool RBAC (Dual-Layer Enforcement)

Two authorization checks on every request. Neither is sufficient alone.

**Discovery filtering** (`tools/list`): Gateway returns only allowed tools. The agent doesn't know unauthorized tools exist.

**Call-time enforcement** (`tools/call`): Full ABAC check (resource attributes, amount, time-of-day) before forwarding.

Policy engine: [Cerbos](https://cerbos.dev) (YAML + CEL, sub-ms decisions). Deny-by-default.

### 3. Stateless Session Management

Redis-backed sessions (Phase 1) with optional token-encoded sessions for horizontal scaling. Set `gateway.session_mode` to `"token_encoded"` to enable.

## Tech Stack

- **Python 3.12+** with full async/await
- **FastAPI + Uvicorn** -- async HTTP server
- **FastMCP** -- proxy primitives, multi-server aggregation, namespace isolation
- **Cerbos** -- policy decision point for tool RBAC (async gRPC)
- **Redis** -- session store, nonce cache, rate limiting
- **cryptography + argon2-cffi** -- credential encryption
- **OpenTelemetry** -- OTLP traces, Prometheus metrics
- **structlog** -- structured JSON logging with credential stripping
